# Logs Specification

**Version**: 5

* * *

## Purpose

This specification defines the directory structure, file formats, and quality requirements for all
log files produced during task execution.

**Producer**: `arf.scripts.utils.run_with_logs` (command logs),
`arf.scripts.utils.capture_task_sessions` (session capture logs), task subagents (step logs), and
research skills (search logs).

**Consumers**:

* **Task subagents** — read previous step logs to understand execution context
* **Verificator scripts** — validate log completeness and format
* **Human reviewers** — audit agent behavior and debug failures
* **Aggregator scripts** — collect execution metrics across tasks

* * *

## Directory Structure

All logs live inside the task folder under `logs/` with four subdirectories:

```text
tasks/<task_id>/logs/
├── commands/                    # Auto-generated by run_with_logs.py
│   ├── 001_20260330T120000Z_ruff-check.json
│   ├── 001_20260330T120000Z_ruff-check.stdout.txt
│   └── 001_20260330T120000Z_ruff-check.stderr.txt
├── steps/                       # One folder per step
│   ├── 001_create-branch/
│   │   └── branch_info.txt
│   ├── 002_check-deps/
│   │   └── deps_report.json
│   ├── 003_init-folders/
│   │   └── folders_created.txt
│   ├── 004_research-papers/
│   │   └── step_log.md
│   └── 007_implementation/
│       └── step_log.md
├── searches/                    # One per internet search query
│   ├── 001_20260330T100500Z_semantic-scholar.json
│   └── 002_20260330T100800Z_google-scholar.json
└── sessions/                    # Session capture outputs
    ├── capture_report.json
    ├── codex__2026__04__05__rollout-<sessionId>.jsonl
    └── claude_code__<project-hash>__<sessionId>.jsonl
```

Each step folder is named `<NNN>_<step-id>` where `NNN` is the zero-padded sequential step number
(from `step_tracker.json`) and `<step-id>` is the canonical step ID from
`arf/specifications/task_steps_specification.md`. The contents of each folder are step-specific —
different steps require different files. See the step verification registry
(`arf/scripts/verificators/step_registry.py`) for exact requirements per step.

The `commands/`, `steps/`, `searches/`, and `sessions/` directories are mandatory for every task.
The `searches/` and `sessions/` directories may remain empty in tasks that do not use those log
types, but the directories themselves must still exist.

## Command Logs

Command logs are **auto-generated** by `arf.scripts.utils.run_with_logs`. The agent does not write
these manually — they are produced as a side effect of wrapping every CLI call through the logging
module.

### Usage

```bash
uv run python -m arf.scripts.utils.run_with_logs --task-id <task_id> -- <command...>
```

### JSON Schema

Each command execution produces one JSON file and two text files (stdout, stderr).

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `spec_version` | string | yes | `"1"` |
| `task_id` | string | yes | Task folder name |
| `command` | string | yes | Full command as a single string |
| `exit_code` | int | yes | Process exit code |
| `duration_seconds` | float | yes | Wall-clock duration as fractional seconds |
| `started_at` | string | yes | ISO 8601 UTC timestamp |
| `completed_at` | string | yes | ISO 8601 UTC timestamp |
| `working_directory` | string | yes | Absolute path of the working directory |
| `stdout_file` | string | yes | Relative path to stdout capture file |
| `stderr_file` | string | yes | Relative path to stderr capture file |
| `stdout_lines` | int | yes | Number of lines in stdout |
| `stderr_lines` | int | yes | Number of lines in stderr |
| `truncated` | bool | yes | Whether stdout/stderr were truncated |

### Example

```json
{
  "spec_version": "1",
  "task_id": "t0003_download_training_corpus",
  "command": "uv run python tasks/t0003_download_training_corpus/download.py",
  "exit_code": 0,
  "duration_seconds": 42.371,
  "started_at": "2026-03-30T14:32:01Z",
  "completed_at": "2026-03-30T14:32:43Z",
  "working_directory": "/Users/dev/my-research-project",
  "stdout_file": "001_20260330T143201Z_uv-run-python.stdout.txt",
  "stderr_file": "001_20260330T143201Z_uv-run-python.stderr.txt",
  "stdout_lines": 142,
  "stderr_lines": 3,
  "truncated": false
}
```

### Naming Convention

```text
<NNN>_<YYYYMMDDTHHMMSSZ>_<command-slug>.json
<NNN>_<YYYYMMDDTHHMMSSZ>_<command-slug>.stdout.txt
<NNN>_<YYYYMMDDTHHMMSSZ>_<command-slug>.stderr.txt
```

* `NNN` — zero-padded 3-digit sequence number (001, 002, ...)
* Timestamp — UTC in compact ISO 8601 format
* Command slug — first 3 tokens of the command, lowercased, non-alphanumeric replaced with `-`,
  truncated to 40 characters

### Truncation

If stdout or stderr exceeds 10,000 lines, only the last 10,000 lines are kept. The `truncated` field
is set to `true` and a note is prepended to the output file.

## Step Logs

Each step gets its own **folder** under `logs/steps/`. The folder name follows the pattern
`<NNN>_<step-id>` where `NNN` is the step order and `<step-id>` is the canonical step ID.

Different steps require different files. The exact requirements are defined in the step verification
registry (`arf/scripts/verificators/step_registry.py`).

### Folder Naming Convention

```text
logs/steps/<NNN>_<step-id>/
```

* `NNN` — zero-padded 3-digit number matching the step order
* Step ID — from `arf/specifications/task_steps_specification.md`

### Steps With `step_log.md`

Steps that involve substantial work (research, planning, implementation, analysis) produce a
`step_log.md` inside their folder. This file uses YAML frontmatter and mandatory sections.

#### YAML Frontmatter

```yaml
---
spec_version: "3"
task_id: "t0003_download_training_corpus"
step_number: 5
step_name: "research-internet"
status: "completed"
started_at: "2026-03-30T10:00:00Z"
completed_at: "2026-03-30T10:45:00Z"
---
```

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `spec_version` | string | yes | `"3"` |
| `task_id` | string | yes | Must match the task folder name |
| `step_number` | int | yes | Must match a step in `step_tracker.json` |
| `step_name` | string | yes | Step ID from the steps specification |
| `status` | string | yes | One of: `"completed"`, `"failed"`, `"skipped"` |
| `started_at` | string | yes | ISO 8601 UTC timestamp |
| `completed_at` | string | yes | ISO 8601 UTC timestamp |

#### Mandatory Sections

* `## Summary` — 1-3 sentences. **Minimum**: 20 words.
* `## Actions Taken` — numbered list. **Minimum**: 2 items.
* `## Outputs` — bullet list of files created/modified.
* `## Issues` — problems or "No issues encountered."

### Steps Without `step_log.md`

Preflight steps (`check-deps`, `create-branch`, `init-folders`) produce lightweight artifact files
instead of a full markdown log:

* `check-deps/` → `deps_report.json`
* `create-branch/` → `branch_info.txt`
* `init-folders/` → `folders_created.txt`

### Example: Research Step Folder

```text
logs/steps/005_research-internet/
    step_log.md          # step log with frontmatter + sections
```

### Example: Preflight Step Folder

```text
logs/steps/002_check-deps/
    deps_report.json     # output of verify_task_dependencies.py
```

### Step Verification

After completing each step, run the step verificator:

```bash
uv run python -m arf.scripts.verificators.verify_step <task_id> <step_id>
```

This checks the step folder contents against the registry requirements for that step type.

## Search Logs

Search logs capture **internet research queries** and their results. One JSON file per search query
executed during the research stage.

### JSON Schema

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `spec_version` | string | yes | `"1"` |
| `task_id` | string | yes | Task folder name |
| `step_number` | int | yes | Step that triggered this search |
| `timestamp` | string | yes | ISO 8601 UTC timestamp |
| `source` | string | yes | Search engine or API name |
| `query` | string | yes | The search query string |
| `results_count` | int | yes | Number of results returned |
| `results` | list[Result] | yes | Top results (max 20) |
| `error` | string/null | yes | Error message if search failed, else `null` |

### Result Object

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `title` | string | yes | Result title |
| `url` | string | yes | Result URL |
| `snippet` | string/null | no | Brief excerpt or abstract |
| `relevance_note` | string/null | no | Why this result matters (optional) |

### Naming Convention

```text
<NNN>_<YYYYMMDDTHHMMSSZ>_<source-slug>.json
```

* `NNN` — zero-padded 3-digit sequence number
* Timestamp — UTC in compact ISO 8601 format
* Source slug — lowercase source name (e.g., `semantic-scholar`, `google-scholar`, `arxiv`)

### Example

```json
{
  "spec_version": "1",
  "task_id": "t0042_bert_baseline_wsd",
  "step_number": 2,
  "timestamp": "2026-03-30T10:05:00Z",
  "source": "Semantic Scholar",
  "query": "word sense disambiguation BERT contextualized embeddings 2025",
  "results_count": 47,
  "results": [
    {
      "title": "Frozen-BERT Baselines for All-Words WSD",
      "url": "https://api.semanticscholar.org/graph/v1/paper/abc123",
      "snippet": "We evaluate frozen BERT representations for WSD using...",
      "relevance_note": "Direct BERT-based WSD approach, highly relevant"
    }
  ],
  "error": null
}
```

## Session Logs

Session logs preserve raw CLI conversation transcripts for audit and debugging. During the
`reporting` step, the session capture utility scans supported CLI transcript roots, copies every
task-matching JSONL transcript into `logs/sessions/`, and writes a machine-readable
`capture_report.json`.

### Source

Supported transcript roots:

```text
~/.codex/sessions/<YYYY>/<MM>/<DD>/<session-file>.jsonl
~/.claude/projects/<project-hash>/<sessionId>.jsonl
```

The capture utility recursively scans these roots, reads each JSONL file, and matches files whose
content contains the task ID.

### Format

Raw JSONL (JSON Lines). Each line is a JSON object representing one event from the underlying CLI
tool. Exact fields vary by product. Common examples include user messages, assistant messages, tool
use requests, tool results, timestamps, working directory metadata, and branch metadata.

### Naming Convention

```text
codex__<relative-source-path>.jsonl
claude_code__<relative-source-path>.jsonl
```

The copied file name preserves the source kind plus the original relative path from that source
root, flattened with `__`. This keeps names unique while retaining enough source context for manual
cross-referencing. Session transcripts exceeding 4 MB are automatically compressed during capture,
producing `.jsonl.gz` files. Both `.jsonl` and `.jsonl.gz` are valid session transcript formats.

### Identification for Parallel Tasks

The capture utility uses a tiered matching strategy to identify session transcripts belonging to a
task:

1. **PID mapping** (Claude Code): Reads `~/.claude/sessions/<pid>.json` files to find sessions whose
   `cwd` matches the task worktree or main repo directory. This is the most precise method and
   avoids false positives from sessions that merely mention the task ID in output.

2. **Thread ID** (Codex): Reads the `CODEX_THREAD_ID` environment variable and matches session files
   whose name contains the thread ID.

3. **Content fallback**: Scans candidate JSONL files for the task ID string. For both Claude Code
   and Codex sessions, a match requires evidence that the session actually ran in the task worktree
   — either the file's on-disk path includes a worktree indicator (task ID in a directory component
   under a `-worktrees` parent), or the JSONL body records a `cwd` field (at any nesting depth)
   whose value references a `-worktrees/<task_id>` directory. This second signal lets the capture
   utility match orchestrator sessions whose Claude Code transcript lives under the
   main-repo-encoded projects directory (because `/execute-task` launches from the repo root) but
   whose subagent tool calls and message envelopes record worktree `cwd` values. Sessions that
   merely mention the task ID in freeform message text, with no matching `cwd` evidence, are
   excluded.

Files matched by precise methods (PID or thread ID) are not re-scanned during the content fallback
phase.

### Capture Report

The `logs/sessions/capture_report.json` file records what the capture utility checked and what it
copied.

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `spec_version` | string | yes | `"1"` |
| `task_id` | string | yes | Task folder name |
| `captured_at` | string | yes | ISO 8601 UTC timestamp |
| `checked_roots` | list | yes | One entry per supported session root |
| `copied_sessions` | list | yes | One entry per copied JSONL transcript |
| `errors` | list | yes | Non-fatal scan or copy problems |

Each `checked_roots` entry contains:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `source_kind` | string | yes | Source identifier such as `codex` or `claude_code` |
| `root_path` | string | yes | Absolute source root that was checked |
| `exists` | bool | yes | Whether that source root existed |
| `candidate_file_count` | int | yes | Number of JSONL files scanned under that root |
| `matched_file_count` | int | yes | Number of JSONL files whose content matched the task ID |

Each `copied_sessions` entry contains:

| Field | Type | Required | Description |
| --- | --- | --- | --- |
| `source_kind` | string | yes | Source identifier such as `codex` or `claude_code` |
| `source_path` | string | yes | Absolute source JSONL file path |
| `copied_path` | string | yes | Relative destination path inside the task folder |
| `matched_by` | string | yes | Match strategy: `pid_mapping`, `codex_thread_id`, or `task_id_content` |

### Scope

The capture utility copies every matching raw transcript file it can find in the supported roots. If
no matching transcript exists, `capture_report.json` still records the scan.

## Verification Rules

### Errors

Errors indicate structural problems that must be fixed.

| Code | Description |
| --- | --- |
| `LG-E001` | `logs/` directory does not exist |
| `LG-E002` | `logs/commands/` directory does not exist |
| `LG-E003` | `logs/steps/` directory does not exist |
| `LG-E004` | A command log JSON is invalid or missing required fields |
| `LG-E005` | A step log is missing YAML frontmatter or mandatory sections |
| `LG-E006` | `task_id` in a log does not match the task folder name |
| `LG-E007` | Step log `step_number` has no matching step in `step_tracker.json` |
| `LG-E008` | A completed step in `step_tracker.json` has no step log |

### Warnings

Warnings indicate quality concerns that should be addressed.

| Code | Description |
| --- | --- |
| `LG-W001` | `logs/searches/` directory does not exist |
| `LG-W002` | A search log JSON is missing optional fields |
| `LG-W003` | Step log section below minimum word count |
| `LG-W004` | Command log has non-zero exit code |
| `LG-W005` | No command logs found |
| `LG-W006` | `step_tracker.json` step is missing `log_file` field |
| `LG-W007` | `logs/sessions/` directory is missing or contains no transcript JSONL files |
| `LG-W008` | `logs/sessions/capture_report.json` is missing or inconsistent |
