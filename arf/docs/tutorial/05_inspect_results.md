# 5. Inspect Results

In [part 4](04_brainstorm_next_tasks.md) you ran a couple of tasks through `/execute-task` and
merged them into `main`. Now read what they produced — not by `cat`-ing files in `tasks/`, but by
opening the dashboard.

ARF projects have a dedicated dashboard at [`overview/`](../../../overview/). It is a tree of
human-readable markdown pages generated from the aggregators: a top-level summary, one page per
task, one page per metric, one page per paper, daily news digests, and more. GitHub renders all of
it natively. This is how researchers and reviewers actually look at the project.

For a fully populated public example, open the
[`overview/` dashboard in GliteTech/research-ace-cefr](https://github.com/GliteTech/research-ace-cefr/tree/main/overview).
It shows the same dashboard structure on a real ARF demo project.

## Step 1: The Dashboard Is Already Up to Date

The `execute-task` skill regenerates the dashboard automatically after merging each task PR. It runs
the [materializer](../../scripts/overview/materialize.py), commits the updated `overview/` files,
and pushes — so by the time a task finishes, the dashboard is already current.

If you need to regenerate manually (e.g., after a correction or a manual merge), run:

```bash
uv run python -m arf.scripts.overview.materialize
```

Then commit and push the result.

## Step 2: Open overview/README.md

Open [`overview/README.md`](../../../overview/README.md) on GitHub. The top of the page is a row of
badges with live counts:

```text
Papers: 134   Datasets: 20   Models: 21   Predictions: 68
Libraries: 7  Answers: 28
News: 10      Tasks: 125     Suggestions: 355   Metrics: 36
Results: 36   Task Types: 17
```

Each badge links to a section. Below the badges, the README shows recently completed tasks, latest
papers, current best metric values, and budget burn. **This is the page you open every morning** to
see where the project stands.

## Step 3: Drill into Your Task

Click **Tasks** in the badge row, or open [`overview/tasks/`](../../../overview/tasks/). You will
see:

* `task_pages/` — one markdown page per task with its name, status, dependencies, results, and
  charts
* `by-status/` — every task grouped by status (`completed`, `in_progress`, `not_started`, etc.)
* `by-date-added/` — chronological view

Open the page for your survey task at
`overview/tasks/task_pages/t0001_survey_image_augmentation_papers.md`. It includes the task
description, the steps run, the papers added, and links to the suggestions it produced. No need to
read raw JSON.

## Step 4: Compare Metric Values Across Tasks

Click **Results** in the badge row, or open
[`overview/metrics-results/`](../../../overview/metrics-results/). One markdown page per registered
metric. Each page lists every task that reported that metric, sorted by value, with links back to
the producing task.

This is the page to open when you want to know "what is my current best on metric X?" or "how does
the latest run compare to the baseline?".

## Step 5: Read the Daily News

Click **News** or open [`overview/news/`](../../../overview/news/). One markdown digest per day,
generated by the [`generate-daily-news`](../../skills/generate-daily-news/SKILL.md) skill. Each
entry covers what completed that day, what it found, what cost, and what comes next. Skim the news
folder weekly to keep your mental model current without re-reading every task.

## Step 6: Browse the Other Sections

The dashboard has the same shape for every entity type:

* [`overview/papers/`](../../../overview/papers/) — every paper with summary, by category, by date
  added
* [`overview/datasets/`](../../../overview/datasets/), [`models/`](../../../overview/models/),
  [`libraries/`](../../../overview/libraries/), [`predictions/`](../../../overview/predictions/),
  [`answers/`](../../../overview/answers/) — same pattern for the other asset types
* [`overview/suggestions/`](../../../overview/suggestions/) — open suggestions grouped by priority
  and kind
* [`overview/costs/`](../../../overview/costs/) — spend by task and by service, against budget

Click around. Every entity links to every related entity. The dashboard is the spine of the project
— almost everything you want to know is one or two clicks away.

## When to Look at Raw Files

Inside a task folder, the raw `results/` files exist for two purposes: they are the source of truth
that aggregators read, and they are useful when debugging a single task locally. For day-to-day work
— reviewing what shipped, picking what to do next, checking budgets, comparing baselines — open the
dashboard.
